🗄 Firestore — persists encounter state across all agents · JSON fallback if GCP unavailable

Order of agent execution during a single patient encounter run.

⚙
Orchestrator initializes encounter
ClinicalOrchestrator creates a new encounter ID, loads patient context from Firestore (or creates a new record), and sets the pipeline phase to "consultation".
ClinicalOrchestratorFirestore lookup / createPatient ID assigned
1
Audio processing — both agents run in parallel
ConsultationAgent and HeARAgent process the same audio file simultaneously. Whisper transcribes speech while HeAR extracts acoustic embeddings. HeAR findings inject into the SOAP Objective before the note is finalised.
ConsultationAgentHeARAgentparallelSOAP Note→ Firestore
2
Vitals ingestion — can run in parallel with Step 1
VitalsMonitorAgent parses the FHIR R4 bundle, checks values against clinical thresholds, and auto-generates an admission note if anomalies are detected.
VitalsMonitorAgentFHIR R4 inputAdmission Note→ Firestore
3
Imaging analysis — needs Step 1 output first
RadiologyAgent fetches the SOAP Note from Firestore and combines it with the medical image before calling MedGemma multimodal. The radiology report is clinically contextual — never a standalone image read.
RadiologyAgentdepends on Step 1Radiology Report→ Firestore
4
Discharge aggregation — needs all prior steps
DischargeAgent pulls all outputs from Firestore and sends them to MedGemma as a single unified prompt to generate the discharge summary and ICD-10 codes.
DischargeAgentdepends on Steps 1+2+3Discharge SummaryICD-10 codes→ Firestore
⚙
Orchestrator closes encounter
Marks encounter complete in Firestore. FHIR write-back to EHR is not yet implemented — a known gap planned before April 30.
